Cours : Créer un serveur MCP branché sur Groq

Public visé : développeurs Node.js / JavaScript. Pas de pré-requis TypeScript.

1. Qu'est-ce qu'un MCP ?

MCP signifie Model Context Protocol. C'est un protocole standardisé par Anthropic qui permet à un client IA (Claude, Codex, ChatGPT-like, etc.) de dialoguer avec un serveur externe pour récupérer des outils (tools), des ressources ou des prompts.

L'idée est simple :

  • Le LLM ne sait rien faire d'autre que générer du texte.
  • Le MCP est une extension qui lui donne des super-pouvoirs : lire un fichier, interroger une base de données, appeler une API, etc.
  • Notre projet fait un MCP qui donne au LLM la capacité d'appeler un autre LLM (chez Groq).
ANALOGIE Un serveur MCP, c'est comme un plug-in pour un LLM. Tu branches ton serveur, et le LLM découvre automatiquement les outils qu'il peut utiliser.
┌─────────────┐ MCP protocol ┌──────────────────┐ HTTPS ┌────────┐ │ Codex / │ ◄───── stdio ────► │ notre serveur │ ────────► │ Groq │ │ Claude │ (JSON-RPC) │ MCP (mcp.js) │ OpenAI │ API │ └─────────────┘ └──────────────────┘ SDK └────────┘

2. Architecture du projet

Le projet est découpé en deux fichiers JavaScript :

FichierRôle
dist/groq.jsClient HTTP vers l'API Groq (liste des modèles, completions, embeddings). Un bon vieux module JS qui exporte des fonctions.
dist/mcp.jsServeur MCP proprement dit. Il importe les fonctions de groq.js et les expose comme des tools MCP.

Cette séparation est importante : on pourrait remplacer groq.js par un client OpenAI, Anthropic ou Mistral sans toucher à la logique MCP.

3. Le client Groq (groq.js)

Groq expose une API compatible OpenAI. On peut donc utiliser le SDK openai en changeant juste l'URL de base.

3.1. Configuration et client

import "dotenv/config";
import OpenAI from "openai";

export const groqConfig = {
    defaultModel:  process.env.DEFAULT_MODEL  ?? "openai/gpt-oss-20b",
    fallbackModels: parseModelList(process.env.FALLBACK_MODELS),
    proxyName:     process.env.OLLAMA_PROXY_NAME ?? "groq",
    groqBaseUrl:   process.env.GROQ_BASE_URL ?? "https://api.groq.com/openai/v1"
};

const groqApiKey = process.env.GROQ_API_KEY;
if (!groqApiKey) {
    throw new Error("GROQ_API_KEY is required.");
}

export const groqClient = new OpenAI({
    apiKey: groqApiKey,
    baseURL: groqConfig.groqBaseUrl
});

À lire ligne par ligne :

  • dotenv/config charge automatiquement les variables du fichier .env.
  • On utilise ?? (nullish coalescing) pour donner des valeurs par défaut.
  • Si la clé API n'est pas définie, on lève une erreur — mieux vaut planter tôt que silencieusement.
  • On instancie le client OpenAI mais en pointant vers api.groq.com : c'est ça, le "proxy" conceptuel.
Rappel JS a ?? b renvoie a sauf si a vaut null ou undefined, auquel cas on renvoie b. Contrairement à a || b, ?? ne traite pas 0 ou "" comme "vide".

3.2. Lister les modèles

export async function listModels() {
    try {
        const response = await groqClient.models.list();
        return response.data.map((model) => ({
            id: model.id,
            owned_by: "groq",
            created: typeof model.created === "number" ? model.created : undefined
        }));
    }
    catch {
        // Si Groq ne répond pas, on renvoie au moins le modèle par défaut
        return [{ id: groqConfig.defaultModel, owned_by: "groq" }];
    }
}

3.3. Créer une complétion (chat)

export async function createChatCompletion(options) {
    const { result } = await tryModels(async (model) => {
        const request = {
            model,
            messages: options.messages,
            temperature: options.temperature,
            max_completion_tokens: options.maxCompletionTokens,
            response_format: options.jsonMode ? { type: "json_object" } : undefined
        };
        return groqClient.chat.completions.create(request);
    }, options.model);

    return result;
}

Cette fonction est le cœur du projet. Elle construit un payload OpenAI standard et appelle Groq. Notez le paramètre response_format qui force la réponse en JSON si jsonMode est activé.

tryModels() est un wrapper qui tente le modèle demandé puis les modèles de fallback si l'appel échoue (429, 500, rate limit, etc.). On reviendra dessus plus bas.

4. Créer le serveur MCP

Maintenant, attaquons dist/mcp.js. Les 8 premières lignes posent toutes les bases :

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import {
    createChatCompletion, createEmbedding, getModel,
    groqConfig, listModels, normalizeContent, toChatMessage
} from "./groq.js";

const server = new McpServer({
    name: "groq-mcp",
    version: "0.1.0"
});

Les 3 dépendances-clés

  • @modelcontextprotocol/sdk — le SDK officiel pour coder un serveur MCP.
  • zod — une librairie de validation de schémas. On l'utilise pour décrire les paramètres que chaque tool accepte.
  • ./groq.js — notre module local.
Pourquoi Zod ? Quand un LLM appelle un tool, il envoie des arguments en JSON. Zod valide et parse automatiquement ces arguments. Si un type est faux, le SDK renvoie une erreur claire au LLM — qui peut alors corriger et réessayer.

5. Enregistrer des outils (tools)

Un tool MCP, c'est une fonction exposée au LLM. Il faut lui donner 3 choses :

  1. Un nom (string)
  2. Une description (titre + description + schéma d'entrée)
  3. Un callback asynchrone qui reçoit les arguments validés et renvoie un résultat.
server.registerTool(
    "mon_tool",
    {
        title: "Mon outil",
        description: "Ce que fait l'outil, en clair pour le LLM.",
        inputSchema: { /* schéma zod */ }
    },
    async (args) => {
        // ... faire le boulot ...
        return { content: [ { type: "text", text: "résultat" } ] };
    }
);
Important Le champ description est lu par le LLM pour décider quand appeler l'outil. Plus elle est précise, plus les appels seront pertinents.

Le projet enregistre 4 tools. Voyons-les un par un.

6. Tool 1 — list_models

Le tool le plus simple : aucun argument d'entrée, renvoie la liste des modèles Groq.

server.registerTool("list_models", {
    title: "List models",
    description: "Lists models available through the configured Groq provider.",
    inputSchema: {}   // aucun paramètre
}, async () => {
    const models = await listModels();
    return {
        content: [
            {
                type: "text",
                text: JSON.stringify({
                    provider: "groq",
                    default_model: groqConfig.defaultModel,
                    models: models.map((m) => m.id)
                }, null, 2)
            }
        ]
    };
});

À retenir :

  • inputSchema: {} — schéma vide = tool sans argument.
  • La valeur de retour respecte toujours le même format : { content: [ { type: "text", text: "..." } ] }.
  • On utilise JSON.stringify(obj, null, 2) pour renvoyer un JSON lisible.

7. Tool 2 — generate_text

Ici, un vrai appel au LLM. L'entrée est plus riche :

server.registerTool("generate_text", {
    title: "Generate text",
    description: "Runs a single-prompt chat completion against the Groq API.",
    inputSchema: {
        model: z.string().optional(),
        prompt: z.string(),                                 // obligatoire
        system: z.string().optional(),
        temperature: z.number().min(0).max(2).optional(),
        max_completion_tokens: z.number().int().positive().optional(),
        json_mode: z.boolean().optional()
    }
}, async ({ model, prompt, system, temperature, max_completion_tokens, json_mode }) => {
    const messages = [];
    if (system) messages.push({ role: "system", content: system });
    messages.push({ role: "user", content: prompt });

    const completion = await createChatCompletion({
        model,
        messages,
        temperature,
        maxCompletionTokens: max_completion_tokens,
        jsonMode: json_mode
    });

    const text = completion.choices[0]?.message?.content ?? "";

    return {
        content: [ { type: "text", text } ],
        structuredContent: {
            model: getModel(model),
            resolved_model: completion.model ?? getModel(model),
            output: text,
            finish_reason: completion.choices[0]?.finish_reason ?? "stop",
            usage: completion.usage ?? null
        }
    };
});

Zoom sur le schéma Zod

ParamètreTypeRègle
modelstringoptionnel — prend defaultModel sinon
promptstringobligatoire
systemstringoptionnel — instruction système
temperaturenumberentre 0 et 2
max_completion_tokensintegerentier positif
json_modebooleanforce la sortie en JSON

Ce que renvoie le tool

Deux champs coexistent dans la réponse :

  • content — le texte "humain" que le LLM client va lire.
  • structuredContent — des métadonnées (modèle utilisé, usage tokens, etc.) que le LLM peut parser si besoin.
Attention completion.choices[0]?.message?.content ?? "" utilise l'optional chaining et ?? pour éviter un plantage si le modèle renvoie une réponse vide. Règle d'or : un serveur MCP ne doit jamais planter — il doit toujours renvoyer quelque chose au LLM.

8. Tool 3 — chat_completion

Variante multi-tour. Au lieu d'un simple prompt, on passe un tableau de messages déjà formatés (comme dans l'API OpenAI).

inputSchema: {
    model: z.string().optional(),
    messages: z.array(
        z.object({
            role: z.enum(["system", "user", "assistant", "tool"]),
            content: z.string().optional(),
            images: z.array(z.string()).optional()
        })
    ).min(1),                     // au moins 1 message
    temperature: z.number().min(0).max(2).optional(),
    max_completion_tokens: z.number().int().positive().optional(),
    json_mode: z.boolean().optional()
}

Dans le callback, chaque message passe par toChatMessage() pour être normalisé au format OpenAI :

messages: messages.map((message) => toChatMessage(message))

Jetons un œil à toChatMessage (défini dans groq.js) :

export function toChatMessage(message) {
    const content = normalizeContent(message);

    if (message.role === "system")    return { role: "system", content };
    if (message.role === "assistant") return { role: "assistant", content };
    if (message.role === "tool")      return { role: "user", content: `[tool]\n${content}` };
    return { role: "user", content };
}

Le rôle "tool" de Ollama n'existe pas chez Groq/OpenAI → on le transforme en "user" avec un préfixe [tool]. Astuce de compatibilité.

9. Tool 4 — embed_text

Les embeddings sont des représentations vectorielles de texte. Utiles pour la recherche sémantique ou le clustering.

server.registerTool("embed_text", {
    title: "Embed text",
    description: "Generates vector embeddings for one or more input strings.",
    inputSchema: {
        model: z.string().optional(),
        input: z.union([z.string(), z.array(z.string()).min(1)]),
        dimensions: z.number().int().positive().optional()
    }
}, async ({ model, input, dimensions }) => {
    const embedding = await createEmbedding({ model, input, dimensions });

    return {
        content: [ { type: "text", text: JSON.stringify({
            model: getModel(model),
            resolved_model: embedding.model ?? getModel(model),
            vectors: embedding.data.length,
            dimensions: embedding.data[0]?.embedding.length ?? 0
        }, null, 2) } ],
        structuredContent: {
            model: getModel(model),
            resolved_model: embedding.model ?? getModel(model),
            embeddings: embedding.data.map((item) => item.embedding),
            usage: embedding.usage ?? null
        }
    };
});

Remarquez z.union([z.string(), z.array(z.string()).min(1)]) : on accepte soit un string seul, soit un tableau non-vide de strings.

10. Le transport stdio

Les deux dernières lignes du fichier sont capitales :

const transport = new StdioServerTransport();
await server.connect(transport);

Un serveur MCP doit parler à son client via un canal de transport. Les deux principaux :

TransportQuand l'utiliser
StdioServerTransportProcessus local : le client lance le serveur en enfant et communique via stdin/stdout. C'est notre cas.
HttpServerTransportServeur distant accessible via HTTP + SSE.
Piège classique En mode stdio, n'utilisez JAMAIS console.log dans votre serveur MCP ! La sortie standard est réservée au protocole. Utilisez console.error (stderr) pour vos logs.

11. Le fallback de modèles

Groq a des quotas. Si votre modèle préféré est saturé, on bascule automatiquement sur un autre. C'est le rôle de tryModels() dans groq.js :

async function tryModels(execute, requestedModel) {
    const models = resolveModelCandidates(requestedModel);
    let lastError = undefined;

    for (const model of models) {
        try {
            const result = await execute(model);
            attachResolvedModel(result, model);
            return { model, result };
        }
        catch (error) {
            lastError = error;
            if (!shouldFallback(error)) throw error;
        }
    }
    throw lastError;
}

La logique :

  1. On construit la liste : modèle demandé + modèles de fallback (sans doublons).
  2. On tente chaque modèle dans l'ordre.
  3. Si une erreur est "récupérable" (rate limit, 5xx, timeout…), on passe au suivant.
  4. Sinon on propage l'erreur (requête mal formée → inutile de réessayer).

shouldFallback() regarde le code HTTP et des mots-clés dans le message d'erreur :

if (status === 429) return true;
if (status >= 500) return true;
// + match sur "rate limit", "quota", "timeout", etc.

12. Brancher le MCP dans Codex

Une fois le serveur buildé, il suffit d'ajouter quelques lignes dans ~/.codex/config.toml :

[mcp_servers.groq]
command = "node"
args = ["C:/chemin/vers/dist/mcp.js"]

Puis vérifier avec :

codex mcp list

Codex va alors lancer node dist/mcp.js en arrière-plan, ouvrir un pipe stdin/stdout vers le processus, et découvrir automatiquement les 4 tools.

13. Exercices pour consolider

Exercice 1 — Pourquoi inputSchema utilise-t-il Zod plutôt que du JSON Schema brut ?

Zod fait à la fois la validation ET le parsing typé. Le SDK MCP convertit automatiquement le schéma Zod en JSON Schema pour l'annoncer au LLM, et valide les arguments reçus avant d'appeler le callback. On gagne en robustesse et en lisibilité.

Exercice 2 — Ajoutez un 5ᵉ tool count_tokens qui prend un text et renvoie un nombre approximatif de tokens (diviser la longueur par 4, par exemple). 
server.registerTool("count_tokens", {
    title: "Count tokens",
    description: "Approximation grossière du nombre de tokens d'un texte.",
    inputSchema: { text: z.string() }
}, async ({ text }) => {
    const approx = Math.ceil(text.length / 4);
    return {
        content: [ { type: "text", text: `~${approx} tokens` } ],
        structuredContent: { tokens: approx, length: text.length }
    };
});
Exercice 3 — Que se passe-t-il si on appelle generate_text sans le champ prompt ?

Zod rejette la requête avant même d'entrer dans le callback. Le SDK MCP renvoie une erreur de validation au LLM, qui peut ré-appeler le tool avec le bon schéma.

Exercice 4 — Pourquoi sépare-t-on groq.js et mcp.js ?

Séparation des responsabilités : groq.js est un client HTTP réutilisable (testable, indépendant). mcp.js l'expose via MCP. On pourrait ajouter un deuxième fichier server.js qui exposerait les mêmes fonctions via Express — c'est d'ailleurs ce que fait le projet complet.

Exercice 5 — En regardant shouldFallback(), pourquoi une erreur 400 (Bad Request) n'est pas traitée comme un cas de fallback ?

Une 400 indique que NOTRE requête est fautive (JSON mal formé, paramètre interdit…). Réessayer avec un autre modèle ne changera rien — il faut corriger le code appelant. On remonte donc l'erreur immédiatement.
